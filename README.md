## 42. Testing Conventions

When writing tests for the Unified AI Workspace, strict isolation rules apply due to Vitest's parallel test file execution and the shared SQLite database:

- **Unique User IDs:** Always use a unique `userId` per test file to prevent state collision. Never hardcode shared static user IDs like `test-user-models`.
- **No Global Cleanup:** Never use global `deleteMany({})` or `updateMany({})` as this will delete records created by other concurrently running tests.
- **No Raw SQL:** Raw SQL execution (`$executeRaw`, `$queryRaw`) and mutative literal SQL strings (`TRUNCATE TABLE`, `DROP TABLE`, etc.) are prohibited in test files.
- **Test Isolation Helper:** Use `withTestUserScope` or `cleanupTestUserData` from `apps/api/src/test/testIsolation.ts` for safe, deterministic cleanup that respects database foreign keys.
- **Cross-user Tests:** If a test involves multiple users, instantiate multiple isolated scopes (`userA`, `userB`) under the same test run.
- **Expected Logs:** Tests that intentionally simulate scheduler, queue, or provider failures should capture and assert expected `console.*` output with `apps/api/src/test/logCapture.ts` instead of leaving noisy logs in normal test output.
- **CI Checks:** A static checker (`pnpm test:isolation`) runs in CI to reject any test files using global mutations or raw SQL. To bypass this for a legitimate reason, add `// test-isolation-allow-global-cleanup: <specific reason>` or `// test-isolation-allow-raw-sql: <specific reason>` on the same or preceding line. The reason must be explicitly provided (at least 10 characters).

## 43. Provider Auto-Recovery Policies

Provider recovery policies are safe automation scaffolds for provider incidents and diagnostics drift alerts. Policies are configured per user and can match trigger type, provider, severity, and status, with a per-policy cooldown to prevent loops.

Implemented triggers:

- `provider_incident_opened`
- `provider_incident_repeated`
- `provider_incident_critical`
- `diagnostics_drift_alert_opened`
- `diagnostics_drift_alert_error`
- `no_usable_models`

Implemented safe actions:

- `notify_in_app`
- `run_safe_health_check`
- `run_safe_ui_diagnostics`
- `create_or_update_incident`
- `mark_provider_temporarily_degraded`
- `prefer_fallback_provider`
- `disable_model_temporarily`

Forbidden actions are rejected by validation:

- auto-login or auto-reconnect
- password or OTP entry
- CAPTCHA, challenge, paywall, or rate-limit bypass
- prompt submission
- raw DOM capture or screenshot capture
- session, cookie, storageState, API key, webhook URL, or signing-secret export

Policy preview and history views never execute actions. Auto-recovery runs store only safe IDs and sanitized metadata. Safe health checks and UI diagnostics reuse the existing provider health and diagnostics paths, do not send prompts, and persist only sanitized diagnostics history.

## 44. Duration-Bound Recovery Overrides

Provider recovery policies can create temporary, reversible overrides for provider and model behavior:

- `mark_provider_temporarily_degraded`: marks a provider degraded for UI warnings and automatic model selection. Mode can be `avoid_if_possible` or `block_for_duration`.
- `prefer_fallback_provider`: temporarily prefers the first usable provider from a configured fallback order for automatic/default selection.
- `disable_model_temporarily`: temporarily disables a model or provider-level model for routing and model selection.

Safety properties:

- Overrides always have `startsAt` and `expiresAt`; duration is bounded from 5 minutes to 7 days.
- Overrides are virtual by default. They do not overwrite `UserModelPreference`, API key model scopes, provider sessions, cookies, or browser storage.
- Rollback is available from `/settings/provider-recovery` and `POST /settings/provider-recovery/overrides/:id/rollback`.
- Expiry is available through `POST /settings/provider-recovery/overrides/expire` for the current user and through the CLI:

```bash
pnpm provider-recovery-overrides:expire -- --dry-run
pnpm provider-recovery-overrides:expire
```

Scheduled expiry cleanup can also run inside the API process. In production-like runtime it defaults to enabled unless explicitly disabled; in local/test it stays disabled unless configured:

```bash
PROVIDER_RECOVERY_OVERRIDE_EXPIRY_SCHEDULER_ENABLED=true
PROVIDER_RECOVERY_OVERRIDE_EXPIRY_INTERVAL_SECONDS=300
PROVIDER_RECOVERY_OVERRIDE_EXPIRY_LOCK_TTL_SECONDS=120
PROVIDER_RECOVERY_OVERRIDE_EXPIRY_MAX_PER_RUN=500
```

The scheduler uses the Redis lock key `provider-recovery-overrides:expiry-lock` with `SET NX EX` so multiple API replicas do not expire the same batch at once. The scheduler only marks past-due active overrides as expired and writes safe audit metadata; it never opens provider UI, submits prompts, reconnects accounts, runs live diagnostics, or changes permanent model preferences. Resolver correctness does not depend on the scheduler: `/v1/models`, `/v1/chat/completions`, internal chat, model settings, and provider health views ignore past-due active overrides even before cleanup runs.

Scheduler observability is available from the provider recovery settings API and UI. The status record stores safe operational metadata only: scheduler name, enabled state, last start/finish timestamps, last status, sanitized error text, lock state, run/failure/skip counters, and aggregate batch counts such as scanned/expired/skipped/duration. It does not store prompt text, provider page content, cookies, localStorage, storageState, API keys, webhook secrets, or raw Redis connection URLs.

Tests that only need model metadata import side-effect-free helpers instead of route modules that initialize queue/Redis resources. Scheduler tests mock the Redis lock path and suppress expected simulated lock warnings so normal `pnpm test` output stays free of Redis `ECONNREFUSED` noise.

Runtime behavior:

- `/v1/models` includes safe `metadata.recovery` fields such as `providerDegraded`, `temporarilyDisabled`, `disabledUntil`, and safe reasons.
- `/v1/chat/completions` rejects a temporarily disabled exact model with OpenAI-compatible error code `model_temporarily_disabled`.
- Internal chat disables provider selection/send when a model is temporarily disabled or a provider is blocked for the duration.
- Views and previews do not execute recovery actions.

## 46. Organization Roles and Admin Boundaries

The workspace has four local roles stored on `User.role`:

- `owner`: full administrative access.
- `admin`: full administrative access.
- `member`: read-oriented workspace access, notification read-state updates, usage/model reads, and diagnostics reads.
- `viewer`: dashboard/read-only access only.

Sensitive settings are protected by backend permission guards. The frontend hides or disables matching controls, but backend authorization is the source of truth.

Permission groups:

- API keys: `apiKeys.read`, `apiKeys.write`
- Provider sessions: `providerConnections.read`, `providerConnections.write`
- Provider diagnostics: `providerDiagnostics.read`, `providerDiagnostics.action`
- Recovery policies and overrides: `providerRecovery.read`, `providerRecovery.write`
- Webhooks: `webhooks.read`, `webhooks.write`
- Notifications: `notifications.read`, `notifications.write`
- Usage analytics: `usage.read`
- Models and provider rate limits: `models.read`, `models.write`
- Release/operator metadata: `release.read`

Boundary notes:

- Existing users and local self-host users default to `owner` for backward compatibility.
- `/health`, `/ready`, `/version`, `/v1/models`, and `/v1/chat/completions` keep their existing behavior.
- Non-admin roles cannot create, rotate, revoke, or scope internal API keys.
- Non-admin roles cannot connect, disconnect, test, or persist provider sessions.
- Non-admin roles cannot update recovery policies, rollback overrides, rotate webhook secrets, run diagnostics actions, or save model/rate-limit settings.
- Permission-denied responses are safe: `{ "error": "permission_denied" }`.
- Permission-denied audit entries store only safe metadata: user ID, permission, route, request ID, and result. They do not store cookies, localStorage, storageState, API keys, webhook secrets, prompts, provider HTML, or session payloads.

## 47. User and Role Management

The Settings area includes a `Users & Roles` page for local workspace user administration.

Access rules:

- `owner`: can list users, review role audit events, and change user roles.
- `admin`: can list users and review role audit events, but cannot change roles in v1.
- `member` and `viewer`: cannot access user management APIs.

Backend permission guards are the source of truth:

- `users.read` protects user list and role audit reads.
- `users.manageRoles` protects role changes and is owner-only in v1.
- Role updates only accept `owner`, `admin`, `member`, or `viewer`.
- No-op role updates return successfully and do not create duplicate audit events.
- The last remaining owner cannot be demoted.
- Owner self-demotion requires explicit confirmation and another owner must already exist.

The user list returns only safe fields: user ID, email, display name, role, created time, and updated time. It never returns password hashes, local auth secrets, provider session blobs, cookies, storageState, API key hashes, webhook secrets, provider HTML, prompts, or tokens.

Role audit events store safe metadata only: actor user ID, target user ID, previous role, next role, action, and timestamp. They do not store emails, names, session payloads, secrets, prompts, or provider content.

### 29. Creating workspaces

The application now supports **Workspace Creation**, allowing users to create their own isolated environment.
- The creator of the workspace is automatically granted the **owner** role.
- New workspaces start completely empty. For maximum security, **no data is copied from other workspaces**. This includes provider sessions, API keys, webhook destinations, and recovery policies.
- After creating a new workspace, the Workspace Switcher securely swaps the user's current context to the new workspace, reloading the UI to guarantee no state is leaked.
- The application still natively supports single-workspace/local deployments. The default 'Local Workspace' will continue to handle users without an explicit workspace assignment.

### 30. Workspace Invites & Email Delivery

The workspace supports **Workspace Invites**, allowing owners to invite new members to their workspace.
- **Secure by Default**: Invite tokens are hashed at rest using SHA-256. Raw tokens are only shown once during creation. The list API never exposes raw tokens or their hashes.
- **Expiry Management**: Invites expire after 7 days by default. A background scheduler automatically cleans up expired invites. Owners can explicitly revoke or manage invites from the Settings area.
- **Role Isolation**: Only `owner` role users can manage invites. New memberships default to `member` role, but owners can specify any valid role.
- **Email Delivery Support**: The system supports multiple email delivery modes via `WORKSPACE_INVITE_EMAIL_PROVIDER` (`noop`, `console_dry_run`, `smtp`). 
- **Delivery Safety Guardrails**: Real SMTP delivery is disabled by default and requires explicit configuration (`WORKSPACE_INVITE_EMAIL_ALLOW_REAL_SEND=true` and `WORKSPACE_INVITE_EMAIL_DRY_RUN=false`). In test environments, real sending is hard-blocked regardless of configuration. Delivery attempts are logged securely for auditing but never include sensitive invite tokens or links. Error messages strictly redact any accidentally exposed provider secrets or API keys.
